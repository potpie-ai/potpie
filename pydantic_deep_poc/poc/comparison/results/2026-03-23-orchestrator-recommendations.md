# Orchestrator Recommendations: celery-to-hatchet PoC

## What failed

The trace shows three separate failures of control, not one:

1. The supervisor explored for too long instead of delegating early.
2. The execute subagent had the wrong tool shape, so it could bypass CCM.
3. The delegated task payload contained almost no discovered context, so the subagent re-explored.

This means the fix is not "better prompt" alone. The architecture must enforce a protocol.

## Diagnosis from current PoC

- The supervisor prompt is mostly a thin wrapper around Potpie's `code_gen_task_prompt`, with no explicit delegation contract or stopping rules.
- The execute subagent is generic and still has access to discovery-style tools.
- `include_skills=False`, so skills are not available at all.
- The current prompt loader does not load `AGENTS.md`/`DEEP.md`; it only injects the Potpie task prompt and a small `context_block`.
- `include_filesystem=False` only affects the main agent. `pydantic-deep` still injects console + todo toolsets into subagents by default.

## Recommended architecture

Use a 4-role architecture with explicit artifacts between stages.

### 1. Orchestrator

Responsibilities:

- classify task complexity
- decide whether the task needs subagents
- create work slices
- pass context packets to workers
- collect results
- trigger final verification

Rules:

- never write code directly
- never do broad exploration after the first discovery phase
- never delegate without a context packet
- never call implementation-only tools

### 2. Discovery Agent

Responsibilities:

- read-only exploration
- identify impacted files
- extract migration patterns and APIs
- produce a compact discovery brief

Allowed tools:

- `fetch_file`
- `fetch_files_batch`
- `get_code_file_structure`
- `analyze_code_structure`
- `web_search_tool`
- `webpage_extractor`
- `bash_command` limited to grep/find style commands

Forbidden:

- any write tool
- CCM tools
- git mutation tools

Output artifact:

- `discovery_brief.md` or structured JSON with:
  - impacted files
  - relevant symbols
  - external API snippets
  - invariants
  - open questions

### 3. Implementer Agent

Responsibilities:

- execute only the assigned slice
- modify code only through CCM
- avoid fresh exploration unless the packet explicitly allows it

Allowed tools:

- CCM tools only
- `fetch_file` only for files named in the packet
- `bash_command` for narrow validation

Forbidden:

- `web_search_tool`
- `webpage_extractor`
- broad file listing
- generic filesystem write tools

Output artifact:

- `slice_result.json` with:
  - files changed
  - rationale
  - tests run
  - unresolved issues

### 4. Verifier Agent

Responsibilities:

- run tests
- check imports and startup paths
- compare discovered scope vs implemented scope

Allowed tools:

- `bash_command`
- `fetch_file`
- `show_diff`

Forbidden:

- new feature edits unless specifically told to patch a failing test

## How to stop subagents from re-entering exploration

Do all four of these, not just one:

1. Split discovery from implementation.
   The current `think_execute` agent mixes both, so the model has no reason to stay in execution mode.

2. Remove research tools from implementers.
   If the execute agent can browse docs and scan the repo, it will.

3. Pass a required context packet in every delegation.
   The packet should include file paths, grep results, discovered APIs, and acceptance criteria.

4. Add a hard prompt rule:
   "Do not perform broad repo exploration. If required context is missing, ask the parent once with a specific question."

## Context packet format

Every `task()` call should pass something like this:

```text
TASK_TYPE: implementation
OBJECTIVE: Replace Celery task bootstrap in worker startup with Hatchet worker bootstrap.
FILES_IN_SCOPE:
- app/celery/worker.py
- app/worker/bootstrap.py
- app/config/settings.py
KNOWN_FINDINGS:
- Current Celery app init happens in app/celery/__init__.py
- Worker entrypoint imports Celery app from X
- Hatchet startup pattern from docs uses Y
CONSTRAINTS:
- Use CCM tools only
- Do not use write_file/edit_file/hashline_edit
- Do not explore outside FILES_IN_SCOPE unless blocked
DONE_WHEN:
- worker starts with Hatchet bootstrap
- old Celery startup path removed or deprecated cleanly
- imports and config remain valid
IF_BLOCKED:
- Ask parent with one precise question
```

This should be generated by the orchestrator, not improvised by the model.

## Parallel tool calling

You need two different kinds of parallelism.

### A. Parallel tool calls inside one model turn

Use a model/provider combination that actually emits multiple tool calls, and pass:

```python
model_settings={"parallel_tool_calls": True}
```

Also set a useful agent concurrency cap so emitted calls are not serialized by the runtime.

This only helps if the model chooses multiple tool calls in one response.

### B. Parallel subagent execution across slices

For large migrations, do not send one huge delegated task. Split into independent slices and launch several async tasks in one response, then wait/synchronize.

Example slices for Celery -> Hatchet:

- bootstrap and worker lifecycle
- task definitions and decorators
- queue/config/env wiring
- deployment/runtime docs and commands

The existing subagent toolset already supports launching several `task()` calls and later `wait_tasks()`. The issue in the trace is that the supervisor never used that path.

## Prompt changes

### Supervisor prompt additions

Add these sections to the supervisor instructions:

```text
## Operating Contract
- You are an orchestrator, not the primary implementer.
- Delegate implementation to THINK_EXECUTE immediately after a bounded discovery pass.
- Never call CCM tools yourself.
- Never delegate without a context packet.
- Prefer parallel delegation when two or more slices are independent.

## Discovery Budget
- Maximum initial discovery: 8 tool calls or 2 turns, whichever comes first.
- After that, either delegate or stop with a blocker.
- Do not read the same area twice.

## Delegation Policy
- Use sync mode for one small code slice.
- Use async mode for 2+ independent slices.
- When delegating, include:
  - files in scope
  - discovered facts
  - acceptance criteria
  - forbidden tools/actions

## Verification Policy
- After implementation, run verification once.
- Do not resume broad exploration during verification.
```

### Implementer prompt additions

```text
## Execution Contract
- You are an implementation worker.
- Do not perform broad repo exploration.
- Operate only on the provided files unless blocked.
- Use CCM tools only for code changes.
- Do not use generic filesystem write tools.
- If required context is missing, ask_parent once with a specific question.
- Return a compact result: files changed, validations run, blockers.
```

## AGENTS.md vs skills

Use both, for different layers.

### Use `AGENTS.md` / `DEEP.md` for repo truth

Put stable project-wide constraints there:

- architecture map
- where workers live
- allowed migration patterns
- "all code changes go through CCM"
- "supervisor never writes"
- test commands
- branching / PR rules

This is good for every run and every task.

### Use skills for reusable procedures

Create skills for:

- `large-code-migration`
- `celery-to-hatchet-migration`
- `parallel-discovery`
- `ccm-only-implementation`
- `verification-and-rollup`

Skills are better for procedural playbooks, templates, and scripts.

### Recommendation

Do not choose one.

- Put invariant repo guidance in `AGENTS.md` or `DEEP.md`.
- Put migration workflows and prompt templates in skills.

In the current PoC, skills are disabled, so they cannot improve results until `include_skills=True` and a skill directory is wired in.

## Concrete PoC changes

1. Replace the single `think_execute` role with three explicit subagents:
   - `discover`
   - `implement`
   - `verify`

2. Add custom subagent descriptions for `task()` so the supervisor sees strict usage guidance in the tool docs.

3. Stop using the default subagent toolset factory from `create_deep_agent` if you need CCM-only implementers.
   Build or patch the subagent toolset path so implementers do not get injected console/filesystem tools by default.

4. Enable:
   - `include_skills=True`
   - `context_discovery=True` or explicit `context_files=[...]`
   - `model_settings={"parallel_tool_calls": True}`
   - checkpoints / context manager for long runs

5. Raise request budgets for migration-class tasks and checkpoint between phases.

6. Emit a machine-readable discovery artifact and require implementers to consume it.

## Minimal target state

If you want the smallest change set with the biggest effect:

1. Strengthen the supervisor prompt with explicit delegation rules.
2. Create a dedicated read-only discovery agent.
3. Create a CCM-only implementer with no web/discovery tools.
4. Pass a structured context packet in `task()`.
5. Switch to a model that reliably uses parallel tool calls and subagent delegation.

That will cut most of the token burn you saw in this trace.
