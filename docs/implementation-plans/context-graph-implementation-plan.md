# Context Graph — Implementation Plan (Phase 1)

This document is a **step-by-step implementation plan** for Phase 1 of the [Context Graph Architecture](./context-graph-architecture-plan.md). It assumes the architecture doc has been read; here we focus on **what to build, in what order, and where**.

---

## Prerequisites

- [Context Graph Architecture Plan](./context-graph-architecture-plan.md) read and agreed.
- **Infra:** A separate Neo4j instance for Graphiti. The existing Neo4j remains for the code graph. Plan for env vars: `GRAPHITI_NEO4J_URI`, `GRAPHITI_NEO4J_USER`, `GRAPHITI_NEO4J_PASSWORD`.
- **Feature flag (recommended):** Env var `CONTEXT_GRAPH_ENABLED=false` so the new code path is off until validated.

---

## Phase 1 scope (reminder)

- Postgres: `context_sync_state` + `context_ingestion_log`.
- Graphiti client: add episodes with `group_id=project_id`, search with `group_ids=[project_id]`.
- Episodes: GitHub PR (opened/merged) and GitHub commit only; default-branch commits optional for first slice.
- ETL: One Celery task to backfill GitHub PRs (and optionally commits) for a project; one-time job to queue all projects.
- Live: Webhook handler extended so GitHub PR merge events enqueue context-graph ingestion (dedup + episode → Graphiti).
- Agent: `get_project_context` tool, registered and added to SpecGenAgent (can be gated by feature flag).
- No Jira/Linear/docs in Phase 1; no custom entity types; no Redis cache.

---

## Implementation order

Steps are ordered by dependency. Complete each step (or logical sub-step) before moving to the next unless marked "can parallelize."

### Step 2: Postgres schema (sync state + ingestion log)

**Goal:** Create the two tables used for ETL progress and deduplication.

**Tasks:**

1. Add SQLAlchemy models in `app/modules/context_graph/models.py` using `app.core.base_model.Base` (NOT `app.core.database.Base` — every ORM model in this project uses `base_model.Base`). **You must set `__tablename__` explicitly** on each model — the `Base` class auto-generates `__tablename__` from `cls.__name__.lower()` (e.g. `ContextSyncState` → `contextsyncstate`), which is not what we want. Every model in the codebase overrides it (e.g. `Project` → `__tablename__ = "projects"`):

   ```python
   from app.core.base_model import Base

   class ContextSyncState(Base):
       __tablename__ = "context_sync_state"
       # ... columns ...

   class ContextIngestionLog(Base):
       __tablename__ = "context_ingestion_log"
       # ... columns ...
   ```

2. Register the new models in `app/core/models.py` (this is where all models are imported for Alembic and Celery to discover them — see existing pattern with `from app.modules.projects.projects_model import Project  # noqa`):


- `app/modules/context_graph/models.py` (new).
- `app/core/models.py` (add import of new models).

### Step 3: Context graph module and Graphiti client

**Goal:** A thin async wrapper around Graphiti that (a) initializes Graphiti with the context-graph Neo4j config, (b) exposes `add_episode(group_id=project_id)` and `search(group_ids=[project_id])`, and (c) respects `CONTEXT_GRAPH_ENABLED`.

**Tasks:**

1. Implement `app/modules/context_graph/graphiti_client.py`:
   - Graphiti is async-first. Expose async methods.
   - **Important API details** (verified against `graphiti-core 0.28.2` source):
     - `add_episode()` requires `source=EpisodeType.text` (imported from `graphiti_core.nodes`). It uses `group_id` (singular string).
     - `search()` uses `group_ids` (plural, a **list** of strings) — not `group_id`. It returns `list[EntityEdge]` (edges only). For nodes+edges+communities, use the advanced `search_()` method.
     - `search()` limit is controlled by `num_results` parameter.

     ```python
     from graphiti_core import Graphiti
     from graphiti_core.nodes import EpisodeType

     class ContextGraphClient:
         def __init__(self):
             cfg = config_provider.get_context_graph_config()
             self._enabled = cfg["enabled"]
             if self._enabled:
                 self._graphiti = Graphiti(
                     cfg["neo4j_uri"], cfg["neo4j_user"], cfg["neo4j_password"]
                 )

         async def add_episode(self, project_id, name, episode_body,
                               source_description, reference_time) -> str:
             if not self._enabled:
                 return ""
             result = await self._graphiti.add_episode(
                 name=name,
                 episode_body=episode_body,
                 source=EpisodeType.text,
                 source_description=source_description,
                 reference_time=reference_time,
                 group_id=project_id,       # singular string
             )
             return str(result.uuid)

         async def search(self, project_id, query, limit=10) -> list:
             if not self._enabled:
                 return []
             return await self._graphiti.search(
                 query=query,
                 group_ids=[project_id],     # plural, list of strings
                 num_results=limit,
             )
     ```
   - **Callers (Celery tasks) must use `self.run_async(coro)` from `BaseTask`** to call async methods (see `app/celery/tasks/base_task.py` — `run_async()` wraps `asyncio.run()`). Do not call `asyncio.run()` directly inside tasks.
   - Agent tool callers can use `asyncio.run()` directly (same pattern as all other tools in `app/modules/intelligence/tools/`).

**Files to create:**

- `app/modules/context_graph/graphiti_client.py`

**Acceptance:** With Graphiti Neo4j running and env set, a short async test script can add one episode and run one search. With `CONTEXT_GRAPH_ENABLED=false`, calls are no-ops.

**Reference:** Architecture doc §2, §6 (Graphiti as-is; group_id).

### Step 4: Episode formatting (GitHub PR and commit)

**Goal:** Turn a GitHub PR (opened or merged) and a GitHub commit into the episode text and `source_id` defined in the architecture.

**Tasks:**

1. Implement pure functions in `app/modules/context_graph/episode_formatters.py`:
   - `format_github_pr_episode(pr_data: dict, event_type: str) -> dict` where `event_type` is `"opened"` or `"merged"`. Returns `{name, episode_body, source_description, source_id}`.
     - Include branch info: `"PR #<n>: <title>. Branch: <head_branch> → <base_branch>. Author: <author>. Files changed: <file list>. Description: <body>."` (see architecture §7).
     - `source_id`: `pr_<n>_opened` or `pr_<n>_merged`.
   - `format_github_commit_episode(commit_data: dict, branch_name: str) -> dict`. Returns same keys.
     - Include: `"Commit <sha>: <message>. Branch: <branch_name>. Author: <author>."`.
     - `source_id`: `commit_<sha>`.
2. The `pr_data` dict shape should match what `GitHubProvider.get_pull_request()` (in `app/modules/code_provider/github/github_provider.py`) returns — keys: `number`, `title`, `body`, `head_branch`, `base_branch`, `author`, `files` (optional). Keep formatters as pure functions (no DB, no network).

**Files to create:**

- `app/modules/context_graph/episode_formatters.py`

**Acceptance:** Given a sample PR dict and commit dict, formatters produce text with branch, title, author, and correct `source_id`.

**Reference:** Architecture doc §7 (Episode format).

### Step 5: Ingestion service (dedup + add_episode + log)

**Goal:** Single entry point: "ensure this episode is in the context graph and logged." Idempotent by design.

**Tasks:**

1. Implement `app/modules/context_graph/ingestion_service.py`:
   - Depends on: async DB session (SQLAlchemy), `ContextGraphClient`, episode formatters.
   - Method: `async ingest_episode(db, project_id, source_type, source_id, payload, event_type)`.
   - Logic:
     1. If `CONTEXT_GRAPH_ENABLED` is false → return (no-op).
     2. Query `context_ingestion_log` for `(project_id, source_type, source_id)`. If row exists → return (idempotent).
     3. Call the right formatter based on `source_type` (`"github_pr"` → `format_github_pr_episode`, `"github_commit"` → `format_github_commit_episode`).
     4. Call `ContextGraphClient.add_episode(project_id, ...)` → get `graphiti_episode_uuid`.
     5. Insert into `context_ingestion_log` and commit.
     6. On Graphiti failure: log error and re-raise (let caller handle retry).
   - DB session must be async (`AsyncSession`) since this will be called from async Celery tasks (via `self.async_db()`) and from the async tool.

**Files to create:**

- `app/modules/context_graph/ingestion_service.py`

**Acceptance:** Calling `ingest_episode` twice with the same `(project_id, source_type, source_id)` hits Graphiti only once; second call is a no-op due to unique index check.

**Reference:** Architecture doc §5 (ingestion log), §8 (write paths).

### Step 6: ETL backfill task (GitHub PRs for one project)

**Goal:** A Celery task that backfills GitHub PRs for a given `project_id` into the context graph.

**Tasks:**

1. Create `app/modules/context_graph/tasks.py` with a Celery task:

   ```python
   @celery_app.task(
       bind=True,
       base=BaseTask,
       name="app.modules.context_graph.tasks.context_graph_backfill_project",
   )
   def context_graph_backfill_project(self, project_id: str) -> None:
       # 1. Sync DB work BEFORE run_async (BaseTask rule):
       project = self.db.query(Project).filter(Project.id == project_id).first()
       if not project or not project.repo_name:
           return
       repo_name = project.repo_name
       branch_name = project.branch_name  # branch_name is stored in the Project model

       # 2. Async work inside run_async:
       self.run_async(_backfill_async(self, project_id, repo_name, branch_name))
   ```

   - `_backfill_async` is an inner async function that:
     - Opens `async with self.async_db() as db:` for DB access.
     - Reads/creates `context_sync_state` row for `(project_id, "github_pr")`. Uses `last_synced_at` as a cursor (fetch PRs merged after that date).
     - Fetches PRs using `GitHubProvider` (via `get_github_app_client(repo_name)` then `repo.get_pulls(state="closed")`). Filter merged PRs.
     - For each PR: call `ingest_episode(db, project_id, "github_pr", f"pr_{pr.number}_merged", pr_data, "merged")`.
     - On completion: update `context_sync_state.last_synced_at` and `status`.
   - Use a dedicated Celery queue: add `"app.modules.context_graph.tasks.context_graph_backfill_project": {"queue": "context-graph-etl"}` to `task_routes` in `celery_app.py`.
   - Batch size: process at most N=100 PRs per run; resume from cursor on next run.

2. **Register the task** — add at the bottom of `app/celery/celery_app.py`:

   ```python
   import app.modules.context_graph.tasks  # noqa
   ```

   This matches the existing pattern of `import app.celery.tasks.parsing_tasks` and `import app.modules.event_bus.tasks.event_tasks`.

**Files to create/change:**

- `app/modules/context_graph/tasks.py` (new).
- `app/celery/celery_app.py` (add import and task_routes entry).

**Acceptance:** Triggering the task for a project backfills PRs into Graphiti and `context_ingestion_log`; re-running does not duplicate; `context_sync_state` reflects progress.

**Reference:** Architecture doc §8.1 (ETL backfill).

### Step 8: Webhook handler — GitHub PR merge → context graph

**Goal:** When a GitHub PR merge webhook is received, enqueue a context-graph ingestion for that merged PR.

**Tasks:**

1. Add a `_process_github_webhook` method to `app/modules/event_bus/handlers/webhook_handler.py`. Currently the handler only handles `linear` and `sentry` — **GitHub is not handled at all**. Add:

   ```python
   # In _process_by_integration_type:
   elif integration_type.lower() == "github":
       return self._process_github_webhook(event_type, payload, integration)
   ```

   and:

   ```python
   def _process_github_webhook(self, event_type, payload, integration):
       action = payload.get("action")
       pr = payload.get("pull_request", {})
       if event_type == "pull_request" and action == "closed" and pr.get("merged"):
           repo_name = payload.get("repository", {}).get("full_name")
           if not repo_name:
               return {"processed": False, "reason": "no repo_name in payload"}
           # Resolve project_id: query Project by repo_name
           # NOTE: multiple projects may share the same repo_name (different branches).
           # For now, enqueue for all matching projects; dedup handles collisions.
           from app.modules.projects.projects_model import Project
           projects = (
               self.db.query(Project)
               .filter(Project.repo_name == repo_name, Project.is_deleted == False)
               .all()
           )
           for project in projects:
               from app.modules.context_graph.tasks import ingest_pr_from_webhook
               ingest_pr_from_webhook.delay(project.id, pr)
       return {"processed": True}
   ```

2. Add `ingest_pr_from_webhook(project_id, pr_payload)` task to `app/modules/context_graph/tasks.py`:
   - Receives the raw `pull_request` dict from the GitHub webhook payload.
   - Builds `source_id = f"pr_{pr_payload['number']}_merged"`.
   - Calls ingestion service `ingest_episode(...)` (via `run_async`).
   - This task is async-safe using the same `BaseTask` + `run_async` pattern.

3. **Note on webhook delivery:** GitHub App webhooks need to be configured to send `pull_request` events to your endpoint. Verify that the GitHub App setup delivers PR events to the app's webhook endpoint. If no endpoint exists for GitHub App webhooks, a new endpoint may need to be added (this is an infra concern, not a code concern, but worth flagging).

**Files to change:**

- `app/modules/event_bus/handlers/webhook_handler.py` (add `_process_github_webhook` method and routing).
- `app/modules/context_graph/tasks.py` (add `ingest_pr_from_webhook` task).

**Acceptance:** Simulating a `pull_request` / `closed` + `merged` webhook for a known repo creates one episode and one ingestion log row. Re-delivery does not duplicate.

**Reference:** Architecture doc §8.2 (Live triggers).

### Step 9: `get_project_context` tool

**Goal:** Agent tool that queries Graphiti with `group_ids=[project_id]` and returns normalized context items.

**Tasks:**

1. Create `app/modules/intelligence/tools/context_tools/__init__.py` (empty) and `app/modules/intelligence/tools/context_tools/get_project_context_tool.py`:

   ```python
   import asyncio
   from typing import List, Optional
   from langchain_core.tools import StructuredTool
   from pydantic import BaseModel, Field

   class GetProjectContextInput(BaseModel):
       project_id: str = Field(description="The project/repository ID (UUID)")
       query: Optional[str] = Field(None, description="What to search for")
       branch: Optional[str] = Field(None, description="Optional branch to bias results")
       file_paths: Optional[List[str]] = Field(None, description="Optional file paths to bias results")
       limit: Optional[int] = Field(10, description="Max number of context items to return")

   class GetProjectContextTool:
       name = "Get Project Context"
       description = "..."  # (full description — see architecture §9)

       def __init__(self, db, user_id):
           self.db = db
           self.user_id = user_id

       async def arun(self, project_id, query, branch, file_paths, limit):
           return await asyncio.to_thread(self.run, project_id, query, branch, file_paths, limit)

       def run(self, project_id, query, branch=None, file_paths=None, limit=10):
           from app.modules.context_graph.graphiti_client import ContextGraphClient
           from app.core.config_provider import config_provider
           if not config_provider.get_context_graph_config()["enabled"]:
               return []
           parts = []
           if branch:
               parts.append(f"branch:{branch}")
           if query:
               parts.append(query)
           if file_paths:
               parts.extend(file_paths)
           search_text = " ".join(parts) or project_id
           client = ContextGraphClient()
           # search() returns list[EntityEdge]; each edge has .fact, .name,
           # .created_at, .source_description, etc.
           edges = asyncio.run(client.search(project_id, search_text, limit or 10))
           return [
               {"fact": e.fact, "name": e.name, "created_at": str(e.created_at)}
               for e in edges
           ]

   def get_project_context_tool(sql_db, user_id) -> StructuredTool:
       tool = GetProjectContextTool(sql_db, user_id)
       return StructuredTool.from_function(
           coroutine=tool.arun,
           func=tool.run,
           name="get_project_context",
           description="...",
           args_schema=GetProjectContextInput,
       )
   ```

   Follow the exact same pattern as `get_ask_knowledge_graph_queries_tool` in `ask_knowledge_graph_queries_tool.py` (sync `run()` uses `asyncio.run()`, async `arun()` uses `asyncio.to_thread()`).

   **Note:** `graphiti.search()` returns `list[EntityEdge]` — edges representing facts. Each `EntityEdge` has fields like `.fact` (string), `.name`, `.created_at`, `.source_description`, `.group_id`. The normalization step converts these into plain dicts for the LLM.

**Files to create:**

- `app/modules/intelligence/tools/context_tools/__init__.py`
- `app/modules/intelligence/tools/context_tools/get_project_context_tool.py`

**Acceptance:** Tool returns context items for a populated project; returns `[]` gracefully when disabled.

**Reference:** Architecture doc §9 (Agent tool).

## Running Phase 1 locally (after implementation)

1. **Neo4j for Graphiti:** Start a second Neo4j instance on a different port (e.g. Docker: `docker run -p 7475:7474 -p 7688:7687 neo4j:latest`). Set `GRAPHITI_NEO4J_URI=bolt://localhost:7688`, `GRAPHITI_NEO4J_USER=neo4j`, `GRAPHITI_NEO4J_PASSWORD=<password>` in `.env`.
