# Apply to Local Workspace - Feature Specification

## Overview

This feature enables users to apply code changes generated by Potpie agents directly to their local VS Code workspace. Users can review changes in the web UI and explicitly choose to apply them to their local filesystem via the VS Code extension.

**Key Requirements:**
- Support multiple concurrent VS Code instances (laptop, desktop, etc.)
- Explicit user consent ("Apply" button click)
- Terminal commands allowed if needed for implementation
- Secure tunnel-based communication between cloud and local workspace

---

## Architecture

### Components

1. **potpie-ui (Next.js Frontend)**
   - Presence detection UI component
   - "Apply to Local Workspace" button
   - Workspace selection (if multiple active)

2. **potpie Backend (FastAPI)**
   - Tunnel status/workspace management endpoints
   - Apply changes orchestration endpoint
   - CodeChangesManager integration

3. **VS Code Extension**
   - Workspace identification
   - Tunnel registration with workspace_id
   - LocalServer file modification endpoints

### Data Flow

```
User (potpie-ui) 
  → Clicks "Apply to Local Workspace"
  → Backend validates tunnel connection
  → Backend retrieves changes from CodeChangesManager
  → Backend routes changes via tunnel to LocalServer
  → Extension applies changes to workspace
  → Extension returns success/failure per file
  → Backend returns results to UI
```

## API Specifications

### Backend Endpoints

#### 1. List Active Workspaces

**Endpoint**: `GET /api/v1/tunnels/workspaces`

**Description**: Returns all active VS Code workspace connections for the authenticated user.

**Authentication**: Required (Bearer token)

**Response**:
```json
{
  "workspaces": [
    {
      "workspace_id": "a1b2c3d4e5f6g7h8",
      "tunnel_url": "https://xyz.trycloudflare.com",
      "connected": true,
      "last_seen": "1704067200",
      "registered_at": "1704060000"
    }
  ],
  "count": 1
}
```

**Status Codes**:
- `200 OK`: Success
- `401 Unauthorized`: Invalid or missing token

---

#### 2. Get Tunnel Status

**Endpoint**: `GET /api/v1/tunnels/status?conversation_id={conversation_id}`

**Description**: Check if user has an active tunnel connection (optionally scoped to conversation).

**Authentication**: Required (Bearer token)

**Query Parameters**:
- `conversation_id` (optional): Scope check to specific conversation

**Response**:
```json
{
  "connected": true,
  "tunnel_url": "https://xyz.trycloudflare.com",
  "conversation_id": "conv_123" // if scoped
}
```

**Status Codes**:
- `200 OK`: Success
- `401 Unauthorized`: Invalid or missing token

---

#### 3. Apply Changes to Local Workspace

**Endpoint**: `POST /api/v1/conversations/{conversation_id}/apply-changes`

**Description**: Apply all pending code changes from CodeChangesManager to the user's local VS Code workspace.

**Authentication**: Required (Bearer token)

**Path Parameters**:
- `conversation_id`: The conversation containing the changes to apply

**Request Body**:
```json
{
  "workspace_id": "a1b2c3d4e5f6g7h8" // Optional: target specific workspace
}
```

**Response**:
```json
{
  "success": true,
  "message": "Applied 3 of 3 file(s)",
  "files_applied": [
    {
      "file_path": "src/utils/helper.ts",
      "status": "applied",
      "change_type": "update"
    },
    {
      "file_path": "src/components/Button.tsx",
      "status": "applied",
      "change_type": "add"
    },
    {
      "file_path": "old_file.ts",
      "status": "failed",
      "error": "File not found"
    }
  ]
}
```

**Status Codes**:
- `200 OK`: Changes applied (may have partial failures)
- `404 Not Found`: No active tunnel connection
- `400 Bad Request`: Invalid conversation_id or request
- `401 Unauthorized`: Invalid or missing token
- `500 Internal Server Error`: Tunnel communication failure

### Phase 2: Extension Changes

#### 2.1 Workspace ID Generation

**File**: `potpie-vscode-extension/src/TunnelManager.ts`

**Changes**:
- Add `generateWorkspaceId()` method
- Use first workspace folder path + machine hostname
- Include workspace_id in tunnel registration

**Implementation**:
```typescript
private generateWorkspaceId(): string {
  const workspaceFolders = vscode.workspace.workspaceFolders;
  if (!workspaceFolders || workspaceFolders.length === 0) {
    return "default";
  }
  const workspacePath = workspaceFolders[0].uri.fsPath;
  const machineId = os.hostname();
  const combined = `${workspacePath}:${machineId}`;
  const hash = crypto.createHash("sha256").update(combined).digest("hex");
  return hash.substring(0, 16);
}
```

## Security Considerations

### Authentication & Authorization

1. **All endpoints require valid Firebase JWT token**
   - Backend validates token via `AuthService.check_auth`
   - Extension validates token before processing LocalServer requests

2. **User-scoped tunnel access**
   - Tunnels are registered per user_id
   - Backend only returns tunnels for authenticated user
   - Extension only accepts requests from authenticated backend

### Consent & Permissions

1. **Explicit user action required**
   - User must click "Apply to Local Workspace" button
   - No automatic application of changes

2. **Workspace isolation**
   - Changes only apply to workspace that registered the tunnel
   - workspace_id prevents cross-workspace contamination

3. **Terminal command safety** (future enhancement)
   - Extension already has terminal command allowlist
   - Can extend to require explicit approval for cloud-initiated commands

### Tunnel Security

1. **HTTPS-only tunnels**
   - Cloudflare tunnels use HTTPS
   - LocalServer validates origin headers

2. **Health checks**
   - Backend verifies tunnel health before applying changes
   - Stale tunnels are automatically invalidated

3. **TTL-based expiration**
   - Tunnel registrations expire after 1 hour
   - Extension refreshes registration periodically

## User Flows

### Flow 1: First-Time Setup

1. User opens potpie-ui chat
2. UI shows "VS Code Not Connected" badge
3. User clicks "Install Extension"
4. User installs extension from marketplace
5. Extension opens sign-in page (`source=vscode`)
6. User authenticates
7. Extension registers tunnel with workspace_id
8. UI detects connection and shows "VS Code Connected"

### Flow 2: Apply Changes

1. User has active chat with pending changes
2. UI shows "VS Code Connected" badge
3. User clicks "Apply to Local Workspace"
4. Backend validates tunnel connection
5. Backend retrieves changes from CodeChangesManager
6. Backend routes changes via tunnel to LocalServer
7. Extension applies changes to workspace
8. UI shows success message with file count
9. User sees changes in VS Code editor

### Flow 3: Multiple Workspaces

1. User has VS Code open on laptop and desktop
2. Both extensions register with different workspace_ids
3. UI calls `GET /tunnels/workspaces`
4. UI shows workspace selector dropdown
5. User selects target workspace
6. Apply operation targets selected workspace

### Flow 4: Connection Lost

1. User closes VS Code
2. Tunnel health check fails
3. Backend marks tunnel as disconnected
4. UI shows "VS Code Not Connected" after next poll
5. "Apply" button becomes disabled
6. User reopens VS Code → connection restored

## Error Handling

### Backend Errors

| Error | HTTP Status | User Message |
|-------|-------------|--------------|
| No tunnel connection | 404 | "No active VS Code connection. Please ensure VS Code extension is running." |
| Tunnel unhealthy | 503 | "VS Code connection is unstable. Please check VS Code and try again." |
| Invalid conversation | 400 | "Invalid conversation ID." |
| Partial apply failure | 200 | "Applied X of Y file(s). Some files failed: [list]" |

### Extension Errors

| Error | Handling |
|-------|----------|
| File not found | Return error in response, continue with other files |
| Permission denied | Return error, log to extension console |
| Invalid file path | Validate before processing, return error |

### Frontend Errors

| Error | User Experience |
|-------|-----------------|
| Network failure | Toast error, retry button |
| Auth token expired | Redirect to sign-in |
| Tunnel disconnected | Disable apply button, show connection status |

## Questions & Decisions

### Open Questions

1. Should we support applying changes to multiple workspaces simultaneously?
   - **Decision**: No, user must select one workspace per apply operation

2. Should we allow partial applies (some files succeed, others fail)?
   - **Decision**: Yes, return per-file results, continue on failure

3. Should we require explicit approval for terminal commands?
   - **Decision**: Yes, extend terminal allowlist to cloud-initiated commands (Phase 2)

4. How should we handle file conflicts (local edits vs. cloud changes)?
   - **Decision**: Cloud changes overwrite local edits (document this clearly to users)

### Design Decisions

1. **Workspace ID Format**: SHA256 hash (first 16 chars) for stability and uniqueness
2. **Apply Scope**: All pending changes in conversation (not per-file granularity initially)
3. **Error Handling**: Continue on failure, return detailed per-file results
4. **Connection Detection**: Polling-based (30s interval) for simplicity
